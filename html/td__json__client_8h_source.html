97 /*
98  * New TDLib JSON interface.
99  *
100  * The main TDLib interface is asynchronous. To match requests with a corresponding response, the field "@extra" can
101  * be added to the request object. The corresponding response will have an "@extra" field with exactly the same value.
102  * Each returned object will have an "@client_id" field, containing the identifier of the client for which
103  * a response or an update was received.
104  *
105  * A TDLib client instance can be created through td_create_client_id.
106  * Requests can be sent using td_send and the received client identifier.
107  * New updates and responses to requests can be received through td_receive from any thread after the first request
108  * has been sent to the client instance. This function must not be called simultaneously from two different threads.
109  * Also note that all updates and responses to requests must be applied in the order they were received for consistency.
110  * Some TDLib requests can be executed synchronously from any thread using td_execute.
111  * TDLib client instances are destroyed automatically after they are closed.
112  *
113  * General pattern of usage:
114  * \code
115  * int client_id = td_create_client_id();
116  * // share the client_id with other threads, which will be able to send requests via td_send
117  *
118  * const double WAIT_TIMEOUT = 10.0; // seconds
119  * while (true) {
120  * const char *result = td_receive(WAIT_TIMEOUT);
121  * if (result) {
122  * // parse the result as a JSON object and process it as an incoming update or the answer to a previously sent request
123  * }
124  * }
125  * \endcode
126  */
